Monitoreo IoT de cámaras de maduración en Almar SRL

Director de tesis: Pablo Navas Torena

Estudiante: Gonzalo Belvisi

Tipo de documento: Informe para Evaluación y Defensa

📋 Resumen Ejecutivo

Problema abordado: Almar SRL requería centralizar el monitoreo de 30 cámaras frigoríficas de maduración de bananas, donde actualmente las lecturas se consultan en interfaces locales (HMI) de forma distribuida y manual, ralentizando la toma de decisiones.

Solución implementada: Plataforma web de monitoreo IoT que integra datos de múltiples fuentes (Sitrad PRO y sensores NUONUO de CO₂ y etileno) en un dashboard único, accesible desde dispositivos móviles y de escritorio, con actualización en tiempo real y sistema de autenticación por roles.

Arquitectura: Sistema de tres capas: (1) Capa física (sensores y controladores), (2) Backend ASP.NET Core 9.0 MVC con SQLite, (3) Dashboard web responsive con Tailwind CSS y jQuery.

Resultados: Prototipo funcional que demuestra valor operativo, reduciendo tiempo de reacción ante desvíos y aportando trazabilidad. Funcionalidades implementadas: visualización en tiempo real, semáforo de temperatura de pulpa, autenticación por roles, modificación de setpoints para operarios autorizados.

1. Objetivo y Alcance del Proyecto

Objetivo Principal

Diseñar e implementar una plataforma web de monitoreo IoT que centralice y visualice la información de todas las cámaras de maduración de Almar SRL, integrando datos heterogéneos (Sitrad PRO y NUONUO) en un dashboard único, accesible y confiable.

1.1 Contexto del Problema

Almar SRL opera 30 cámaras frigoríficas para la maduración de bananas (aproximadamente 250.000 unidades por cámara). El proceso requiere monitoreo constante de variables críticas:

  • Temperatura de aire y temperatura de pulpa: Variable crítica que indica el estado interno de la fruta
  • Humedad relativa: Controla el ambiente de maduración
  • Estado del evaporador: Equipo de climatización
  • Niveles de CO₂ y etileno: Gases que influyen en el proceso de maduración

Situación actual: Las lecturas se consultan en interfaces locales (HMI) de forma manual y distribuida, lo que ralentiza la toma de decisiones y dificulta construir un histórico confiable para análisis y ajustes del proceso.

1.2 Alcance del Proyecto

Incluye:

  • Integración con APIs existentes (Sitrad PRO y NUONUO Refrigeration)
  • Desarrollo de dashboard web responsive con actualización en tiempo real
  • Sistema de autenticación y autorización por roles (Operario/Visualizador)
  • Semáforo de temperatura de pulpa con umbrales configurables
  • Modificación de setpoints para operarios autorizados
  • Despliegue on-premises con exposición segura a Internet (NGROK)

No incluye (trabajo futuro):

  • Buffer local completo para cortes de red prolongados
  • Historial completo con agregaciones avanzadas (mínimos, máximos, promedios)
  • Sistema de alertas externas (correo, mensajería)
  • Reportes exportables (PDF, Excel)

2. Arquitectura de la Solución

Arquitectura de Tres Capas

La solución se estructura en tres capas claramente separadas, facilitando mantenibilidad, escalabilidad y evolución futura.

2.1 Capa de Sensado y Control (Capa Física)

Componentes:

  • 30 Cámaras frigoríficas: Cada una con capacidad de ~250.000 bananas
  • Sensores internos Sitrad: Temperatura de aire, temperatura de pulpa, humedad relativa, estado del evaporador
  • 60 Sensores NUONUO: 30 para CO₂ y 30 para etileno (2 sensores por cámara)
  • Controladores Sitrad: Concentran señales de sensores y las exponen vía Sitrad PRO
  • HMI (Human-Machine Interface): Interfaz local y concentrador edge para sensores NUONUO
  • Gateway PUSR (USR-TCP232-304): Puente RS485 ↔ Red para conectar sensores al cloud del proveedor

Flujo de datos: Sensores físicos → Controladores/HMI → RS485 → Gateway PUSR → Cloud del proveedor (NUONUO) o Sitrad PRO API

2.2 Capa de Adquisición y Lógica de Negocio (Backend)

Implementación: ASP.NET Core 9.0 MVC ejecutándose en servidor local (on-premises)

Componentes principales:

  • SitradApiClient: Consulta periódica a Sitrad PRO API (/instruments, /instruments/{id}/values, /instruments/{id}/functions)
  • NuonuoApiClient: Consume API del proveedor para CO₂ y etileno, vinculando sensores por alias/IMEI
  • Servicios de dominio: Combinan lecturas, aplican reglas de negocio (semáforos, umbrales), implementan tolerancia a fallos
  • AuthService: Gestión de usuarios, autenticación SHA256, validación de códigos de invitación, control de roles
  • AppDbContext (Entity Framework Core): Persistencia en SQLite local (tabla Users, tabla Measurement planificada)

Controladores MVC:

  • SitradController: Dashboard principal, actualización de datos, modificación de setpoints
  • AccountController: Login, registro, logout, gestión de sesiones

2.3 Capa de Presentación (Dashboard Web)

Tecnologías:

  • Razor Views: Renderizado server-side (Index.cshtml, Login.cshtml, Register.cshtml)
  • Tailwind CSS: Diseño responsive mobile-first
  • jQuery 3.6.0: Actualización automática cada 5 segundos mediante AJAX

Funcionalidades:

  • Visualización de tarjetas por cámara con todas las variables
  • Semáforo de temperatura de pulpa (amarillo 17°C, naranja 18°C, rojo ≥19°C)
  • Interfaz adaptativa según rol (Operario: controles habilitados, Visualizador: solo consulta)
  • Notificaciones toast para feedback de acciones

2.4 Justificación de la Arquitectura

  • Separación de responsabilidades: Reduce acoplamiento, facilita evolución independiente
  • Escalabilidad: Backend como punto único de integración, fácil agregar cámaras/sensores
  • Mantenibilidad: Organización clara en controladores, servicios y modelos
  • Seguridad: Autenticación centralizada, exposición controlada vía HTTPS
  • Reutilización: Contrato HTTP/JSON permite futuras apps móviles nativas

4. Desafíos Técnicos y Resoluciones

4.1 Heterogeneidad de Fuentes de Datos

Desafío: Integrar datos de Sitrad PRO (temperatura, humedad, evaporador, setpoints) con NUONUO (CO₂, etileno), cada uno con su propia API y modelo de datos.

Resolución: Modelo de dominio unificado (CameraViewModel), clientes de API específicos que encapsulan diferencias, normalización de unidades y campos antes de construir la vista.

4.2 Restricciones de Red y Acceso a Sitrad

Desafío: Restricciones de red por infraestructura de terceros (Númina), dificultando acceso estable a la API y exposición para pruebas remotas.

Resolución: Identificación de puertos/endpoints críticos, estrategia de acceso que minimiza dependencias, uso de túnel HTTPS (NGROK) para acceso externo controlado.

4.3 Coordinación con Proveedor NUONUO

Desafío: Alta y mapeo de sensores en cloud del proveedor requería coordinación manual (WeChat), con riesgos de demoras e inconsistencias.

Resolución: Esquema explícito de identificación (alias/IMEI ↔ cámara ↔ channel), validación sensor por sensor, procedimiento estandarizado de alta en Intelligent Security Cloud Platform.

4.4 Integración Física y Parámetros de Comunicación

Desafío: Conectar 60 sensores vía RS485, configurar HMI y gateway PUSR, asegurar parámetros alineados (baud rate, timeouts, ciclos).

Resolución: Configuración y estabilización de parámetros en HMI (modo transferencia, frecuencia, baud rate, timeouts), validación extremo a extremo en etapas, consolidación del pipeline completo.

4.5 Cortes Eléctricos y Despliegue On-Premises vs. Nube

Desafío: Cortes eléctricos ocasionales, decisión entre despliegue local o en la nube.

Resolución: Enfoque híbrido on-premises porque: (1) durante apagones, las cámaras también se detienen, no hay datos relevantes que levantar, (2) preferencia operativa del cliente por control local, (3) costo/beneficio favorable, (4) mantenimiento de posibilidad de exposición controlada para futuras integraciones.

4.6 Seguridad y Exposición hacia Internet

Desafío: Acceso desde fuera de red interna sin comprometer seguridad.

Resolución: Servicio en red interna, exposición vía NGROK (HTTPS extremo a extremo), autenticación con usuarios/contraseñas (SHA256), sesiones HTTP-only, control de acceso por roles con código de invitación para operarios.

4.7 Usabilidad del Dashboard

Desafío: Interpretación rápida del estado de cámaras sin sobrecarga visual.

Resolución: Iteraciones de diseño (v1 fondo claro, v2 fondo oscuro), semáforo intuitivo de temperatura de pulpa, diseño mobile-first, tarjetas por cámara con información priorizada.

4.8 Gestión de Usuarios y Persistencia

Desafío: Sistema de autenticación robusto, simple de desplegar, sin servidores externos.

Resolución: SQLite con Entity Framework Core, hash SHA256, código de invitación configurable, usuario admin automático, validación de permisos en servidor y cliente.

6. Preguntas Potenciales de Defensa

❓ Pregunta 1: ¿Por qué elegiste ASP.NET Core sobre otras tecnologías como Node.js o Python?

Respuesta sugerida: ASP.NET Core fue elegido por varias razones técnicas y prácticas: (1) Framework maduro y robusto con excelente soporte para aplicaciones web empresariales, (2) Integración nativa con Entity Framework Core para acceso a datos, (3) Modelo MVC bien estructurado que facilita la organización del código en controladores, servicios y modelos, (4) Soporte multiplataforma permite ejecutar en Windows o Linux según necesidades, (5) Performance excelente para este tipo de aplicaciones, (6) Amplio ecosistema y documentación de Microsoft. Además, dado que Almar SRL utiliza el ecosistema Microsoft, ASP.NET Core facilita futuras integraciones.

❓ Pregunta 2: ¿Por qué SQLite y no una base de datos más robusta como PostgreSQL o SQL Server?

Respuesta sugerida: SQLite fue elegido por simplicidad de despliegue y porque se alinea perfectamente con los requisitos del proyecto: (1) No requiere servidor de base de datos externo, reduciendo complejidad de infraestructura, (2) Archivo local facilita backups y mantenimiento, (3) Ideal para despliegue on-premises donde se prefiere control local, (4) Suficiente para el volumen de datos actual (usuarios y futuras lecturas históricas de 30 cámaras), (5) Entity Framework Core soporta migración a otras bases de datos si fuera necesario en el futuro. La decisión priorizó simplicidad operativa sin comprometer funcionalidad.

❓ Pregunta 3: ¿Por qué decidiste despliegue on-premises en lugar de cloud (AWS, Azure, etc.)?

Respuesta sugerida: La decisión fue estratégica considerando el contexto operativo: (1) Disponibilidad durante cortes eléctricos: Cuando hay apagones, las cámaras y su instrumentación también se detienen, por lo que mantener la app en la nube no agregaría valor durante esos períodos, (2) Preferencia del cliente: Almar SRL manifestó preferencia por mantener control local de la infraestructura, (3) Costo/beneficio: Disponibilidad de equipos en planta y evitar costos recurrentes de infraestructura cloud, (4) Seguridad y control: Datos sensibles del proceso de maduración bajo control directo, (5) Flexibilidad futura: La arquitectura permite exposición controlada de APIs para integraciones futuras sin comprometer el núcleo local. Es un enfoque híbrido que balancea control, costo y operatividad.

❓ Pregunta 4: ¿Cómo garantizas la seguridad del sistema, especialmente la exposición a Internet vía NGROK?

Respuesta sugerida: La seguridad se implementa en múltiples capas: (1) Autenticación: Sistema de usuarios con contraseñas hasheadas (SHA256), no almacenadas en texto plano, (2) Autorización por roles: Separación clara entre Operarios (pueden modificar) y Visualizadores (solo consulta), con código de invitación para registro de operarios, (3) Sesiones seguras: Cookies HTTP-only, timeout de 20 minutos, validación en cada solicitud, (4) HTTPS extremo a extremo: NGROK proporciona túnel HTTPS seguro, cifrando toda la comunicación, (5) Validación en servidor y cliente: Permisos verificados tanto en controladores como en la interfaz, (6) Control de acceso: Solo usuarios autenticados pueden acceder al dashboard, redirección automática a login. La exposición vía NGROK es temporal y controlada, ideal para acceso remoto sin configuración compleja de firewall/NAT.

❓ Pregunta 5: ¿Qué pasaría si la API de Sitrad PRO o NUONUO deja de responder? ¿Cómo maneja el sistema los fallos?

Respuesta sugerida: Se implementaron varias estrategias de tolerancia a fallos: (1) Reintentos exponenciales: Las llamadas a APIs tienen mecanismos de reintento con backoff exponencial para manejar fallos temporales, (2) Marcadores de integridad: El sistema diferencia entre lectura válida, fuera de rango o faltante, permitiendo al usuario entender el estado de los datos, (3) Manejo graceful de errores: Si una API no responde, el dashboard sigue mostrando las variables disponibles de otras fuentes, (4) Buffer local planificado: Se diseñó (aunque no está completamente implementado) un buffer local para almacenar lecturas cuando la conexión se cae y reenviarlas al restablecerse, (5) Feedback visual: El dashboard muestra claramente cuando hay problemas de conectividad mediante toasts y estados visuales. Esta es un área de mejora continua, especialmente el buffer local completo.

❓ Pregunta 6: ¿Cómo escalaría este sistema si Almar SRL quisiera agregar más cámaras o sensores?

Respuesta sugerida: La arquitectura fue diseñada pensando en escalabilidad: (1) Backend como punto único de integración: Para agregar cámaras, solo se ajustan los clientes de API o el modelo de datos, el contrato HTTP/JSON con el dashboard se mantiene, (2) Arquitectura modular: Los servicios (SitradApiClient, NuonuoApiClient) están desacoplados, facilitando agregar nuevos proveedores o sensores, (3) Base de datos: SQLite puede manejar el volumen esperado, y Entity Framework facilita migración a bases más robustas si fuera necesario, (4) Modelo de datos flexible: El CameraViewModel puede extenderse con nuevas variables sin afectar la lógica existente, (5) Separación de capas: Cada capa puede evolucionar independientemente. Por ejemplo, se pueden agregar sensores en la capa física sin modificar el dashboard, siempre que se mantenga el contrato del backend.

❓ Pregunta 7: ¿Por qué usaste SHA256 para las contraseñas y no algoritmos más modernos como bcrypt o Argon2?

Respuesta sugerida: SHA256 fue elegido como balance entre seguridad y simplicidad para este contexto: (1) Suficiente para el caso de uso: SHA256 proporciona seguridad adecuada para este tipo de aplicación empresarial interna, (2) Simplicidad de implementación: Más simple de implementar y mantener que bcrypt/Argon2, especialmente considerando el alcance del proyecto, (3) Estándar ampliamente usado: SHA256 es un estándar reconocido y aceptado, (4) Contexto de uso: Sistema interno con acceso controlado, no es un sistema público masivo, (5) Evolución futura: La arquitectura permite migrar a algoritmos más robustos si fuera necesario sin rehacer todo el sistema. Reconozco que para sistemas públicos de mayor escala, bcrypt o Argon2 serían preferibles por su resistencia a fuerza bruta, pero para este contexto SHA256 es apropiado.

❓ Pregunta 8: ¿Qué métricas o KPIs usaste para validar que el sistema cumple con los objetivos?

Respuesta sugerida: La validación se realizó principalmente de forma cualitativa y funcional: (1) Validación funcional: Todas las variables requeridas se muestran correctamente en el dashboard, (2) Feedback del cliente: Demostraciones con jefes de maduración y operarios mostraron que el sistema facilita la detección rápida de desvíos comparado con HMI locales, (3) Latencia: Actualización cada 5 segundos es aceptable para el proceso de maduración (que no requiere milisegundos), (4) Usabilidad: Interfaz intuitiva validada mediante pruebas informales con usuarios reales, (5) Disponibilidad: Sistema operativo durante ventanas operativas. Métricas cuantitativas más formales (como tiempo promedio de detección de desvíos, reducción de tiempo de consulta) serían valiosas para una evaluación más completa, pero están fuera del alcance de este prototipo inicial.

❓ Pregunta 9: ¿Qué aprendiste del proceso de integración con proveedores externos (NUONUO)?

Respuesta sugerida: La integración con NUONUO fue una experiencia muy valiosa que me enseñó varias lecciones: (1) Importancia de la coordinación: Trabajar con proveedores externos requiere comunicación clara y procedimientos estandarizados, especialmente cuando hay diferencias de huso horario, (2) Validación temprana: Es crítico validar sensor por sensor que el mapeo (alias/IMEI ↔ cámara) sea correcto desde el inicio, (3) Documentación y trazabilidad: Mantener esquemas explícitos de identificación evita errores costosos, (4) Flexibilidad técnica: Adaptarse a las limitaciones y procesos del proveedor (alta manual en cloud, comunicación vía WeChat) mientras se mantiene calidad técnica, (5) Habilidades blandas: Comunicación efectiva, paciencia y organización fueron tan importantes como las habilidades técnicas. Esta experiencia me preparó mejor para proyectos reales donde no todo es controlable internamente.

❓ Pregunta 10: ¿Qué harías diferente si tuvieras que empezar el proyecto de nuevo?

Respuesta sugerida: Si tuviera que empezar de nuevo, haría varios ajustes: (1) Prototipo más temprano con el cliente: Involucrar usuarios finales desde el inicio para validar diseño y funcionalidades, (2) Documentar APIs desde el inicio: Crear documentación clara de las APIs externas y sus contratos, facilitando mantenimiento futuro, (3) Testing más estructurado: Implementar pruebas automatizadas, especialmente para integraciones con APIs externas, (4) Métricas desde el inicio: Definir KPIs cuantitativos para medir impacto real del sistema, (5) Buffer local desde el inicio: Priorizar el buffer local para cortes de red como funcionalidad core, no como trabajo futuro, (6) Documentación técnica más detallada: Más diagramas de flujo, documentación de decisiones arquitectónicas, (7) Iteraciones más pequeñas: Sprints más cortos con feedback más frecuente. Sin embargo, estoy satisfecho con la arquitectura general y las decisiones técnicas principales, que fueron sólidas.

7. Conclusiones y Trabajo Futuro

Conclusiones Principales

El proyecto de monitoreo IoT de cámaras de maduración en Almar SRL demuestra que es factible integrar datos heterogéneos de múltiples fuentes en una plataforma unificada, proporcionando valor operativo inmediato a través de visualización centralizada y accesible.

7.1 Logros del Proyecto

  • Integración exitosa: Se logró integrar datos de Sitrad PRO y NUONUO en un modelo unificado
  • Arquitectura sólida: Diseño de tres capas que facilita mantenimiento y evolución
  • Valor demostrado: Prototipo funcional que ya aporta valor a la operación diaria
  • Aprendizaje integral: Experiencia técnica y de gestión de proyecto con cliente real

7.2 Limitaciones y Trabajo Futuro

Limitaciones actuales:

  • Buffer local para cortes de red aún en fase de diseño
  • Historial completo y agregaciones parcialmente implementadas
  • Sistema de alertas externas no implementado
  • Reportes exportables pendientes

Evolución futura:

  • Completar funcionalidades planificadas (buffer, historial, alertas, reportes)
  • Integración con sistemas corporativos (ERP, Microsoft ecosystem)
  • Analítica predictiva para optimización del proceso
  • Ajustes automáticos de setpoints basados en reglas de negocio
  • Escalado a más cámaras o nuevas variables

7.3 Reflexión Personal

Este proyecto en Almar SRL significó mi primer acercamiento "de punta a punta" a una necesidad operativa real, donde cada decisión técnica tenía impacto directo en la toma de decisiones diaria. Me permitió aprender no solo sobre IoT e integración de sensores, sino también sobre comunicación con usuarios no técnicos, trabajo con personas, iteración rápida a partir del feedback, y priorización de disponibilidad y accesibilidad por encima de "lo ideal" en un laboratorio.

Trabajar con cámaras de maduración me hizo valorar la importancia de la precisión, ya que estos datos influyen en decisiones operativas críticas (por ejemplo, definir si un lote está en condiciones de venta), y la necesidad de transformar datos dispersos en información accionable.

En resumen, fue una experiencia que consolidó mis habilidades técnicas y, al mismo tiempo, reforzó mi motivación por construir soluciones que generen valor concreto en procesos productivos.
